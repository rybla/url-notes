Why LLMs Can't Really Build Software - Zed Blog
llmsoftware engineeringmental modelsaiprogrammingai limitationsautonomous agentscognitionsoftware development
  • Core Thesis: LLMs cannot effectively build non-trivial software because they are incapable of creating, maintaining, and comparing the two distinct mental models essential for engineering: the model of the requirements and the model of what the code actually does.
  • Effective Software Engineering Loop:
    1. Build a mental model of requirements.
    2. Write code to implement the model.
    3. Build a mental model of the code's actual behavior.
    4. Identify discrepancies and iterate on code or requirements.
  • LLM Failures in the Loop:
    • LLMs cannot reliably perform step 3 or 4. They struggle to form an accurate model of the code they've written and get confused when tests fail, guessing whether to fix the code or the tests.
    • They lack the ability to "zoom" in and out of context levels or to temporarily stash a problem's context to solve a sub-problem, unlike human engineers.
  • Identified Technical Weaknesses:
    • Context Omission: Models fail to acquire relevant context that is not explicitly provided.
    • Recency Bias: Model outputs are disproportionately influenced by the most recent information in the context window.
    • Hallucination: Models generate plausible but incorrect details.
  • Conclusion: While useful for generating code for simple, well-defined tasks, LLMs currently function as a tool, not an autonomous engineer. The human developer remains responsible for driving the development process by managing the core mental models and verifying the output.

What Does Palantir Actually Do?
palantirb2bsaassoftwaredata analysisbusiness intelligencegovernmentlaw enforcementcommercialenterprisedata integrationno-codeaiartificial intelligencegothamfoundryapolloaip
  • Core Business Model: Palantir is a Business-to-Business (B2B) Software-as-a-Service (SaaS) company. It does not broker, mine, or own data; it sells software that processes data owned by its clients.
  • Primary Function: The software provides a unified analysis layer that sits on top of a client's existing, often disparate and legacy, data systems. It integrates this data for analysis without requiring a fundamental overhaul of the underlying architecture.
  • Key Products:
    • Gotham: An investigative platform for government and law enforcement clients. It integrates various data sources (e.g., crime reports, booking logs, subpoenaed information) to map networks, centralize intelligence on individuals, and enable complex queries based on specific characteristics.
    • Foundry: A data operations platform for commercial clients. It helps businesses manage supply chains, monitor manufacturing, and track orders by operationalizing their internal data.
    • Apollo: A continuous delivery system that manages and deploys updates for Foundry and Gotham.
    • Artificial Intelligence Platform (AIP): A suite of AI tools that can be integrated into the Gotham and Foundry platforms.
  • User Interface: Designed for non-technical users, allowing them to perform complex data analysis and generate insights without writing code.
  • Deployment Model: Employs "forward deployed software engineers" who act as consultants embedded within client organizations to customize data pipelines and support the software.

Rethinking CLI interfaces for AI
aillmcliapi designagent-based systemshuman-computer interactionsystem designinformation architecturedeveloper toolsai experienceaixshell scriptingsoftware engineering
  • Core Problem: Current CLI and API designs are inefficient for LLM agents, leading to excessive tool calls and context window overflow, particularly with local models.
  • API Design for Agents:
    • APIs should be designed to guide LLMs toward using high-level, convenient functions over low-level, verbose ones.
    • A temporary mitigation is embedding explicit guidance in docstrings to influence function selection.
  • Agent Interaction with CLIs:
    • Agents frequently fail by executing commands in the wrong directory or attempting to bypass quality control hooks (e.g., git commit --no-verify).
    • System-level enforcement (e.g., git wrappers blocking --no-verify, file permissions preventing hook modification) is necessary to keep agents compliant.
  • Proposed Solution: Information Architecture for LLMs:
    • Augment standard CLI tools to provide more contextual information to agents.
    • Example 1: Wrap head to cache output, structure it, and report the total number of lines, preventing resource-intensive re-execution.
    • Example 2: Enhance the command_not_found shell handler to report the current directory ($PWD) and suggest potential correct directories.
  • Future Direction:
    • Develop a suite of LLM-native CLI tools or a custom shell environment for agents.
    • Establish a new discipline, "AI Experience" (AIX), to formalize the principles of designing information systems for non-human users.
